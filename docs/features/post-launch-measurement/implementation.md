# Post-Launch Measurement Implementation

## What Was Added

Post-launch measurement now stores launched experiment data locally and evaluates behavior against the thresholds generated by payment-test specs.

Implemented components:

- SQLite migration `003_post_launch_measurement_tables`.
- Tables: `experiments`, `experiment_events`, `measurement_snapshots`, `experiment_decisions`.
- Repository helpers in `src/db/repositories/experiments.ts`.
- Measurement modules for event recording, metrics aggregation, threshold evaluation, and decision reports.
- CLI entrypoint: `npm run measurement`.
- Markdown/JSON artifacts under `artifacts/ideas/<idea-id>/`.
- Tests using temp SQLite databases and local CSV fixtures only.

## How To Run

Create an experiment from the latest stored payment-test report for an idea:

```bash
npm run measurement -- --idea-id 1 --create
```

Import manual behavior events:

```bash
npm run measurement -- --experiment-id 1 --events ./data/events.csv
```

Evaluate an experiment and persist a report:

```bash
npm run measurement -- --experiment-id 1 --evaluate
```

Import and evaluate in one command:

```bash
npm run measurement -- --experiment-id 1 --events ./data/events.csv --evaluate
```

Use the latest experiment for an idea:

```bash
npm run measurement -- --idea-id 1 --latest --evaluate
```

Record one manual event:

```bash
npm run measurement -- --experiment-id 1 --event-name cta_click --source manual --session-id s1
```

## CSV Format

```text
event_name,occurred_at,source,session_id,metadata_json
page_view,2026-07-07T10:00:00.000Z,manual,s1,{}
cta_click,2026-07-07T10:01:00.000Z,manual,s1,{}
```

Supported events:

- `page_view`
- `pricing_view`
- `cta_click`
- `preview_start`
- `preview_complete`
- `checkout_start`
- `payment_click`
- `email_submit`
- `reply_received`
- `refund_requested`
- `support_contact`

## Decision Behavior

The evaluator parses the text thresholds generated by `payment-test` and classifies each threshold as:

- `strong_signal`
- `weak_signal`
- `kill_signal`
- `inconclusive`

Low-sample, empty, or click-only weak data produces `inconclusive`. A `build_mvp` decision requires the stored strong threshold to be met. A `kill` decision requires the stored kill threshold to be met.

## Non-Goals Preserved

This implementation does not add:

- Real payment processing.
- Search Console OAuth.
- Google Analytics.
- Email automation.
- External analytics calls.
- Public analytics SDK hosting.
